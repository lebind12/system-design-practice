# 챕터 진행 워크플로우

매 학습 세션(= 한 챕터)은 아래 **11단계**를 순서대로 진행합니다. 각 단계의 산출물은 세션 종료 시 노션 페이지와 챕터 `README.md`에 정리됩니다.

## 단계 0 — 챕터 브랜치 생성

디렉토리 생성이나 파일 수정에 **앞서서** 브랜치를 먼저 만듭니다. 한 세션 = 한 챕터 = 한 브랜치 원칙입니다.

```bash
# 1. 현재 위치 확인 (반드시 main 이어야 함)
git status -sb

# 2. main 최신화는 사용자에게 요청 (필요 시)
#    git pull

# 3. 챕터 브랜치 생성 및 체크아웃
git checkout -b chapter/<chapter-name>
```

브랜치 이름 규칙: `chapter/<kebab-case-chapter-name>`. 챕터 디렉토리명과 동일한 이름을 사용합니다 (예: `chapter/url-shortener`).

## 단계 0-b — 책 내용 정리 (구현 착수 전 필수)

어떤 구현 코드도 작성하기 전에, 해당 챕터가 다루는 책의 내용을 먼저 정리합니다. 책을 읽기만 했을 때는 흘려보낸 의사결정 포인트를 글로 고정해두는 단계이며, 이후 MVP 범위 확정(단계 4)의 근거가 됩니다.

정리에 들어가기 전, **반드시 사용자에게 아래 세 가지를 질문**하여 합의합니다.

1. **어디에 정리할지** — 선택지 예시
   - (a) 챕터 디렉토리 내부 파일 (`README.md` 의 설계 섹션 또는 별도 `NOTES.md`)
   - (b) 노션 페이지에 선행 정리
   - (c) 대화창에서만 정리하고 파일 산출물은 만들지 않음
2. **정리 범위** — 책의 해당 장 전체인지, 특정 섹션 위주인지
3. **정리 스타일** — 책 요약 / 핵심 의사결정 포인트 / Q&A 형식 / 이들의 조합

합의된 방식으로 책 내용을 정리한 뒤, **사용자 검토와 피드백을 받은 후에만** 단계 1(요구사항 정의) 로 넘어갑니다. 이 단계를 건너뛰고 구현을 시작하지 않습니다.

> 파일로 산출하는 경우, 권장 위치는 `<chapter>/NOTES.md` 입니다. 챕터 `README.md` 는 실제 구현·벤치·회고 수치를 담는 용도로 비워둡니다.

## 단계 1 — 요구사항 정의 (~10분)

사용자와 함께 아래를 합의해 챕터 `README.md`의 "요구사항" 섹션에 기록합니다.

- **기능 요구사항** (핵심 기능 3~5개)
- **비기능 요구사항** (가용성, 지연시간, 일관성 수준)
- **명시적 비범위** (이 실습에서 다루지 *않을* 것)

## 단계 2 — 개략적 규모 추정 (~5분)

책의 2장 기법을 사용해 수치를 만듭니다.

- 일일 활성 사용자 (DAU)
- 초당 요청(QPS), 피크 QPS
- 저장 용량 (1년 기준)
- 대역폭

추정 수치는 이후 벤치마크 목표가 됩니다.

## 단계 3 — 상위 설계 (~15분)

컴포넌트 다이어그램을 mermaid로 챕터 `README.md`에 그립니다. 최소 포함 요소:

- 클라이언트 / API 게이트웨이 / 서비스 / 데이터 저장소 / 캐시 / 큐
- 각 컴포넌트 간 호출 방향과 프로토콜

## 단계 4 — MVP 범위 확정 (❗가장 중요)

**책 전체 설계를 다 만들면 절대 안 됩니다.** 아래 규칙을 따릅니다.

- **MVP 1개** — 단일 프로세스 + 필수 인프라 1~2종으로 돌아가는 최소 구동
- **확장 실험 2~3개** — MVP 이후 하나씩 붙이며 비교할 변경점
  - 예: 캐시 추가, 샤딩, 복제, 알고리즘 교체, 메시지 큐 분리

이 합의가 끝나야 코드 작성에 들어갑니다.

## 단계 5 — 구현

- `_template/` 을 복사해 챕터 디렉토리를 만듭니다: `cp -r _template <chapter-name>`
- `docker-compose.yml` 의 `name:` 필드를 **디렉토리명과 동일하게** 설정
- **`.env` 파일 생성** — `cp .env.example .env` 후 실제 값 채우기. `.env` 는 커밋되지 않으며, `.env.example` 은 구조만 남겨 공개됩니다.
- **챕터 `README.md` 의 "환경 변수 (.env)" 표 업데이트** — 이 챕터에서 실제로 사용한 변수만 남기고, 각 변수의 의미·예시·사용처를 채워 포트폴리오를 읽는 사람이 구조를 이해할 수 있게 합니다.
- 언어·프레임워크 관례에 맞춰 프로젝트 골격 생성
- Makefile의 `run` / `test` / `bench` 타겟을 실제 명령으로 채움
- 외부 포트 대역은 다른 챕터와 겹치지 않게 선택 후 README 포트 맵에 기록

## 단계 6 — 확장 실험

합의된 실험을 **하나씩** 적용하며 각각 벤치마크를 돌립니다. 각 실험마다:

1. 변경 내용 1~2줄 요약
2. 가설 (무엇이 어떻게 개선될 것 같은가)
3. 실제 결과 수치
4. 가설과의 차이 및 해석

## 단계 7 — 벤치마크

- 도구: `k6`, `wrk`, `hey`, `locust` 중 상황에 맞는 것
- 측정 지표: **throughput(RPS), latency p50/p95/p99, 에러율**
- 조건: 동시 연결 수, 지속 시간, 워밍업 포함 여부를 명시
- 결과는 챕터 `README.md` 의 "벤치마크" 섹션 표에 정리

## 단계 8 — 회고

챕터 `README.md` 의 "회고" 섹션에 기록:

- **배운 것** — 책을 읽기만 했을 때는 안 보였던, 구현하면서 발견한 것
- **막힌 지점과 해결** — 근본 원인 중심으로
- **의사결정 트레이드오프** — 왜 A가 아닌 B를 선택했는가
- **다음에 시도할 것** — 시간이 더 있었다면 해볼 확장

## 단계 9 — 기록 및 마무리

순서대로 진행합니다. 모두 사용자 승인 후 실행합니다.

1. **챕터 브랜치에 최종 커밋** — 의미 단위 커밋은 세션 진행 중 여러 번 쌓여도 됩니다. 마지막에 누락된 파일이 없는지 `git status` 로 확인.
2. **노션 기록** — `docs/NOTION_TEMPLATE.md` 규약에 따라 지정 DB에 페이지 생성. 생성된 노션 페이지 URL 은 다음 단계의 README 인덱스 갱신에 사용됩니다.
3. **루트 `README.md` 인덱스 갱신** — 해당 챕터 행을 완료 상태(✅)로 업데이트하고 디렉토리·언어·노션 링크 채우기. 이 변경도 챕터 브랜치에서 커밋합니다.

## 단계 10 — `main` 병합

모든 문서·코드 변경이 챕터 브랜치에 커밋된 후에만 병합합니다.

```bash
# 1. main 으로 이동
git checkout main

# 2. --no-ff 병합 (챕터 경계를 머지 커밋으로 보존)
git merge --no-ff chapter/<chapter-name> -m "merge: chapter/<chapter-name> into main"

# 3. 병합이 깨끗하면 로컬 브랜치 삭제
git branch -d chapter/<chapter-name>
```

`--no-ff` 옵션은 반드시 사용합니다. fast-forward 로 병합하면 챕터 단위의 경계가 히스토리에서 사라져 포트폴리오 가독성이 떨어집니다.

**원격 푸시는 Claude 가 수행하지 않습니다.** 병합 후 사용자에게 `git push origin main` (또는 `git push --force-with-lease` 가 필요한 상황이라면 그 이유와 함께) 실행을 요청하고, Claude 자신은 푸시하지 않습니다.
