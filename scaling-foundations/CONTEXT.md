## 사건 순서

### 0. 초기 가설과 설계

- lab1 의 목적: 책 1장의 "단일 → DB 분리 → LB + 복제 → 캐시" 진화 경로를 4 stage Docker Compose 로 실제 구동하고 hey 로 부하를 걸어 비교
- 초기 가설 (H1): stage1 이 가장 느리고 단계를 진행할수록 RPS 가 올라갈 것 — 책의 서사대로
- 실제 (c=50): **전 단계에서 역방향**. stage1 19k → stage4 7k RPS. 단일 머신에서 박스를 추가하는 건 오버헤드일 뿐이라는 것이 첫 발견
- 수정 가설 (H2): "동시성을 올리면 단일 JVM 이 포화되어 stage3/4 가 역전할 것"
- 다음 실험: c=500, c=1000 추가

- stage4 가 stage3 보다 항상 에러가 더 많음 (4428 < 11594, 8099 < 17110)

### 2. 1차 가설 — nginx upstream keepalive 누락

에러가 nginx 경유 경로에만 집중 + "서버가 연결을 먼저 끊음" 패턴 →
upstream 커넥션 풀 소진이 의심. 당시 `nginx.conf` 는 다음과 같이 미니멀:

```nginx
events {}

http {
  upstream backend {
    server app1:8080;
    server app2:8080;
  }

  server {
    listen 80;
    location / {
      proxy_pass http://backend;
      proxy_set_header Host $host;
      proxy_set_header X-Forwarded-For $remote_addr;
    }
  }
}
```

**문제 진단** (세 겹):

1. `events { worker_connections 4096; }` 부재 — 기본값 512. c=1000 에서 워커당 한계 도달
2. `upstream` 에 `keepalive 64` 부재 — 모든 요청이 새 TCP 연결을 열고 닫음 → `TIME_WAIT` 누적 → 소켓 고갈 → RST/FIN
3. `proxy_http_version 1.1` + `proxy_set_header Connection ""` 부재 — keepalive 가 있어도 HTTP/1.0 로 떨어지면 매 요청 후 끊김

"stage4 의 에러가 stage3 보다 더 많은 이유" 도 이 진단으로 설명: Redis 왕복이 요청당 평균 처리시간을 늘려 nginx 가 upstream 커넥션을 더 오래 붙잡음 → 풀이 더 빨리 소진.

### 3. 1차 수정 — nginx.conf 3지시어 추가

`lab1-horizontal-scaling/nginx.conf` 를 다음으로 변경:

```nginx
events { worker_connections 4096; }

http {
  upstream backend {
    server app1:8080;
    server app2:8080;
    keepalive 64;
  }

  server {
    listen 80;
    location / {
      proxy_pass http://backend;
      proxy_http_version 1.1;
      proxy_set_header Connection "";
      proxy_set_header Host $host;
      proxy_set_header X-Forwarded-For $remote_addr;
    }
  }
}
```

각 지시어가 왜 필요한지 파일 상단에 주석으로 명기 (미래의 자신이 다시 안 밟게).

### 5. 2차 분석 — "이상하게 느려진" 현상의 정체

첫 반응은 "고쳤는데 더 느려졌다" 였지만, 곱씹어보면 정상이다. 이전의 "8,096 RPS" 는 **암묵적 backpressure** 였다:

- nginx 가 4,428 개 요청을 EOF 로 거부
- hey 의 입장에서 이 거부는 "즉시 실패 → 즉시 재시도" 로 카운트
- 성공한 요청은 큐에 서지 않고 통과 (그래서 평균 지연시간이 적당히 보임)
- **결과적으로 c=500 동시성이 실제로 500 만큼 backend 큐에 서지 않았다**
- RPS 수치는 "일부 요청이 빠르게 실패하는 것 + 성공한 요청의 처리량" 의 합이므로 **정상적인 throughput 이 아니라 실패+통과 혼합**

수정 후에는 **모든 500 개 연결이 진짜로 backend 큐에 서서 처리를 기다린다.** 이제야 진짜 병목이 드러난다.

### 6. 3차 발견 — HikariCP 연결 풀이 새 병목

`application.yml` 확인:

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
```

replica 1개당 **20 개 DB 커넥션만 유지**. stage3/4 는 replica 2 개 = 총 40 개 병렬 DB 작업.

c=500 부하에서 nginx 가 round-robin → 각 replica 당 ~250 동시 요청 → **230 개는 HikariCP 내부 큐에서 대기**.

지연시간 분포가 이 가설을 뒷받침:

| 백분위 | stage3 | stage4 |
|---|---|---|
| p50 | 60 ms | 83 ms |
| p75 | 186 ms | 238 ms |
| p95 | 622 ms | 519 ms |

p50 은 "풀에서 바로 커넥션을 얻은 행운아", p95 는 "큐에서 오래 기다린 사람". 고전적 **커넥션 풀 포화 시그니처** (bimodal distribution).

### 7. c=500 관찰 — stage4 가 stage3 를 처음으로 "미세하게" 이김

| stage | c=500 RPS | p95 |
|---|---:|---:|
| 3 | 2,953 | 622 ms |
| 4 | **3,068** | **519 ms** |

4% 더 빠른 RPS, 17% 낮은 p95. **c=500 한정, Redis 가 처음으로 도움이 되는 구간.**

해석 (당시): Redis 캐시 hit → HikariCP 큐를 건너뜀 → DB 커넥션 경쟁 감소 → 약간의 여유.

이 해석을 "Redis 의 가치는 느린 저장소를 가리는 것이 아니라 하위 의존성을 아끼는 것" 으로 일반화해 기록. **c=1000 에서 이 일반화가 뒤집힘 (아래 9 번 참조).**

### 9. c=1000 에서의 역전 — "어드미션 컨트롤" 가설

**c=500 에서 stage4 > stage3 였는데, c=1000 에서 stage3 > stage4 로 뒤집혔다.** 같은 시스템, 같은 데이터, 동시성만 달라졌는데 우열이 반전.

#### 가설 A — 커넥션 풀은 제약이 아니라 "어드미션 컨트롤" 이다

이 가설이 이번 챕터의 가장 중요한 발견이다.

- **stage3 (HikariCP 20 × 2 replica = 40)**: 풀은 **암묵적 레이트 리미터**. 1000 개 요청이 와도 동시에 DB 를 치는 건 40 개. 나머지는 HikariCP 큐에서 FIFO 대기. **DB 는 항상 여유 있게 돌아감** → 개별 쿼리 자체는 빠르게 완료 → 풀 반환 → 다음 요청 진입
- **stage4 (+redis)**: Redis 경로에는 **어드미션 컨트롤이 없음**. 1000 스레드가 동시에 Lettuce (Spring Boot 기본 Redis 클라이언트) 를 통해 Redis 에 매달림. Lettuce 는 단일 공유 커넥션 + netty 이벤트 루프 구조지만, **앞단의 스레드 경쟁 + future dispatch + 콜백 오버헤드** 가 포화 상황에서 직접 측정 가능한 비용
- 추가: Tomcat 기본 `max-threads=200` × 2 replica = 400 스레드. c=1000 의 600 요청은 Tomcat accept queue 대기. stage3 에서는 어차피 HikariCP 풀에서 또 대기해 큰 차이 없음. **stage4 에서는 "Tomcat queue → Lettuce 경쟁 → Redis 단일 스레드 응답 dispatch" 전 구간 포화**

**직관에 반하는 결론**: **커넥션 풀은 "제약" 이 아니라 "보호 장치" 다.** 한도가 있는 풀은 포화 시에도 DB 에 일정한 부하만 전달해 개별 작업을 빠르게 완료시키고, 큐는 FIFO 로 조용히 풀린다. 제약이 없는 경로는 포화를 그대로 받아 전 구간이 뭉개진다.

#### 가설 B — Redis 자체의 오버헤드

- Redis 는 단일 스레드 서버. 두 replica 에서 초당 ~5,200 Redis ops → 한도 근처는 아니지만 Docker bridge 네트워크 왕복 + Lettuce 직렬화 가 매 요청 누적
- stage3 의 postgres 는 shared buffer hit 로 개별 쿼리 자체는 sub-ms. "네트워크 왕복 + 직렬화" 비용은 stage4 와 거의 동일

즉, Redis 단독의 고정 비용이 "HikariCP 큐 대기 시간" 보다 짧다는 보장은 없다. 특히 포화 상황에선.

#### 가설 C — c=500 의 "stage4 미세 승리" 가 사실 우연이었을 가능성

c=500 에서 stage4 가 stage3 를 3~4% 이겼던 건 측정 분산 안일 수 있다. c=500 과 c=1000 결과를 나란히 놓으면 **stage3/4 는 본질적으로 동일한 throughput 상한 (~3k RPS)** 에 막혀 있으며, 차이는 미세한 큐 동역학 의 산물로 보인다.

**최종 해석**: stage3 과 stage4 는 이 워크로드에서 본질적으로 **같은 병목** (downstream 에 매달린 400+ 스레드의 경합) 을 다르게 드러내고 있을 뿐이며, c=500 의 "stage4 승리" 를 일반화하면 안 된다.

### 10. 직관에 반하는 종합 요약

전 동시성에서 stage1 이 항상 최고:

| stage | c=50 RPS | c=500 RPS | c=1000 RPS |
|---|---:|---:|---:|
| 1 HashMap | 19,012 | 20,430 | **19,600** |
| 2 +pg | 13,447 | 15,527 | 14,382 |
| 3 +LB+복제 | 8,282 | 2,953 | 2,709 |
| 4 +redis | 7,031 | 3,068 | 2,592 |

c=1000 기준 stage1 은 stage4 의 **7.6 배 RPS**. 책 1장의 "단일 → 외부DB → LB+복제 → 캐시" 진화 경로는 이 로컬 단일 머신 조건에서 **단 한 번도 개선으로 나타나지 않았다.** 모든 "개선 단계" 는 실제로는 성능 하락. 이 결론은 반직관적이지만 정직한 측정 결과다.

단, 이것은 **"책이 틀렸다" 는 뜻이 아니다**. 책의 진화 경로는 **단일 머신이 트래픽을 감당 못할 때** 의미가 있다. 우리는 단일 머신이 c=1000 에서도 에러 0 으로 버티는 상황에서 그 다음 단계를 강제했고, 그것이 오버헤드로 나타났을 뿐이다.

### 11. 보조 실험 — HikariCP pool 20 → 100 (어드미션 컨트롤 검증)

가설: "pool 크기가 큰 병목이 아니라 어드미션 컨트롤이다. 키우면 p95 가 악화될 것."

**결과 (c=1000)**:

| stage | baseline (pool 20) RPS | tuned (pool 100) RPS | Δ RPS | baseline p50 | tuned p50 | baseline p95 | tuned p95 |
|---|---:|---:|---|---:|---:|---:|---:|
| 2 (단일 replica) | 14,382 | **17,411** | **+21%** ⬆ | 59.0 | 52.1 ⬇ | 107.5 | **78.2** ⬇ |
| 3 (LB+복제) | 2,709 | 2,642 | -2.5% ≈ | 144.6 | **178.4** ⬆ | 1126.8 | 1083.6 ≈ |
| 4 (+redis) | 2,592 | **2,850** | +10% ⬆ | 211.9 | **248.6** ⬆ | 1078.7 | **856.4** ⬇ |

#### 가설 판정: **조건부 지지 (부분 수정 필요)**

- **stage2 가 가설의 반례**: 단일 replica 에서는 pool=20 이 진짜 하드 리밋이었다. pool=100 으로 키우자 RPS +21%, p95 -27%. "풀 = 어드미션 컨트롤" 이라는 일반화는 틀렸다
- **stage3 은 가설을 강하게 지지**: 풀을 5배 키웠는데 throughput 그대로 (2709 → 2642, -2.5%). 더 결정적인 건 p50 이 **144ms → 178ms 로 악화**. 이건 bimodal → spread 전환의 정확한 signature
- **stage4 는 혼합**: tail (p95) 은 -21% 개선, median (p50) 은 +17% 악화. stage3 과 같은 패턴이지만 Redis 가 섞여서 해석이 복잡

#### 수정된 가설

- **원 가설**: 풀은 제약이 아니라 보호 장치다
- **수정 가설**: 풀 크기가 downstream 처리 능력과 **비슷하거나 크면** 어드미션 컨트롤, **명백히 작으면** 단순 병목. 어느 쪽인지는 **벤치 없이 알 수 없다**.
- stage2: pool=20 ≪ downstream 용량 → 단순 병목. 키우니 개선
- stage3: pool=40 (20×2) ≈ downstream 용량 → 어드미션 컨트롤. 키워도 효과 없음
- stage4: 비슷한 패턴 + Redis 경로의 간접 효과

#### stage3 의 진짜 병목은 뭔가

풀을 5배 키워도 throughput 이 그대로라는 건 **stage3 의 본질적 상한 ~2,700 RPS** 가 pool 바깥에 있다는 뜻. 후보:

1. Postgres 내부 처리 (락·버퍼·쿼리 플래너 오버헤드)
2. **Docker bridge 네트워크** — 모든 패킷이 단일 bridge 통과 (가장 유력)
3. **호스트 CPU 경쟁** — M-series Mac 에서 두 replica container 가 같은 물리 코어 공유
4. Lettuce / Tomcat 내부 경합

**이 실험의 한계**: 로컬 단일 머신 벤치는 "진짜 독립된 replica" 가 아니다. 네트워크·CPU 가 호스트 레벨에서 공유되기 때문에, 책 1장이 가정하는 "원격 분산 환경의 수평 확장" 을 온전히 재현하지 못한다. 이건 "책이 틀렸다" 가 아니라 **우리 측정 환경의 근본적 제약**.

### 12. 복잡도와 튜닝 효과의 역관계

반직관적 관찰: **가장 단순한 stage2 가 pool 튜닝으로 가장 큰 개선 (+21%)** 을 봤다. 더 복잡한 stage3 은 개선 없음.

**일반 원칙**: *"복잡한 시스템은 튜닝이 어렵다. 병목이 여러 곳에 분산돼 있어 한 가지 튜닝으로 큰 이득을 볼 수 없다."*

이건 "마이크로서비스 아키텍처의 관측·운영 비용이 왜 높은가" 의 실험적 증거의 작은 버전이다. 박스가 많아질수록 병목이 분산되어 각각이 약해지고, 전체를 끌어올리려면 **모든 박스를 동시에 튜닝** 해야 한다.

## 배운 것 (노션 "배운 것" 섹션 후보)

### 1. "단일 서버가 버틸 수 있는 워크로드에 스케일 아웃을 하는 것은 오버헤드일 뿐"

c=50 에서 stage1 19k RPS 는 전혀 포화되지 않았다. 그 상태에서 postgres·nginx·redis 를 덧붙이는 건 요청 경로에 네트워크 홉을 추가하는 행위에 불과. 책 1장이 보여주는 진화 경로는 **필요해진 다음에만 의미가 있다.**

### 2. "스케일 아웃의 첫 병목은 앱이 아니라 인프라 설정"

단일 JVM 은 c=1000 까지 에러 0 으로 선형 확장. 그 다음 단계로 넘어가자마자 **nginx upstream keepalive 없음** 으로 무너졌다. 책의 "로드밸런서" 섹션은 박스를 그릴 뿐, 그 박스 내부 튜닝 없이는 동작하지 않는다는 걸 말하지 않는다.

### 3. "에러율이 섞인 벤치 수치는 throughput 이 아니다"

8,096 RPS (에러 1.8%) → 2,953 RPS (에러 0%). **실제 정상 처리량은 후자.** 에러를 "일부 실패" 로 해석하지 말고 "측정 무효" 로 취급해야 한다. 자동 경고 하네스로 영구화.

### 4. "병목은 이동한다. 한 번 풀면 다음 병목이 바로 드러난다"

nginx 풀 → HikariCP 풀 → (다음은 postgres max_connections?). 스케일 아웃은 **"다음 병목이 어디일지 미리 알고 함께 튜닝하는 게임"** 이지, 박스만 추가하는 게임이 아니다.

### 5. "캐시의 진짜 역할은 의존성 아끼기"

Redis 는 "postgres 보다 빨라서" 도움이 되지 않았다 (로컬 postgres 는 shared buffer hit 로 이미 1ms 미만). 대신 **HikariCP 큐 대기를 건너뛰게 해서** 간접적으로 3% 이득. 이 관점은 책 1장의 "캐시는 느린 저장소 앞에" 서술과 결이 다르다.

### 6. "기본값은 벤치 전에 전부 의심하라"

`HikariCP maximum-pool-size: 20` (Spring Boot 기본) 은 c=500 에 명백히 작다. 벤치 시작 전 `.yml`/`.conf` 의 모든 "max", "pool", "worker", "connection" 필드를 훑고 각 값이 부하 시나리오에 맞는지 확인하는 습관이 필요.

### 7. "커넥션 풀은 제약이 아니라 어드미션 컨트롤이다"

c=1000 에서 stage3 (HikariCP 20 × 2) 이 stage4 (Redis 경로) 보다 **더 빠르고 더 안정적** 이었다. 이건 처음엔 "작은 병목" 으로 보였던 HikariCP 풀이, 포화 상황에서는 **DB 를 과부하로부터 보호하는 장치** 로 작동했기 때문이다.

명시적 한도가 있는 시스템은 포화 시에도 지연이 bimodal (빠른 / 대기) 로 갈라져, 한 쪽은 여전히 빠르게 완료된다. 한도가 없는 경로 (stage4 의 Redis call) 는 모든 요청이 고르게 느려지며 multimodal 로 퍼진다.

**실무 교훈**: 풀 크기를 "더 크면 더 좋다" 로 접근하면 안 된다. 풀은 **보호 장치** 다. 적절한 크기는 "DB 가 감당 가능한 동시 작업 수" 에 맞춰야 하며, 그 이상은 큐에서 기다리는 것이 **낫다**. "어드미션 컨트롤" 이 없는 서비스 호출은 포화 상황에서 지연 분포가 무너진다.

### 8. "Redis 도 클라이언트측 경쟁점이 된다"

Lettuce (Spring Boot Redis 기본 클라이언트) 의 단일 공유 커넥션 모델은 "빠른 경로" 로 알려져 있지만, c=1000 수준에서는 Lettuce 앞단의 스레드 경쟁 + future/콜백 dispatch 오버헤드가 직접 측정 가능한 비용이 된다. "Redis 는 빠르다" 는 명제는 클라이언트 코드가 합리적 수준에서 공유/경쟁할 때만 참이다.

### 9. "모드가 바뀌면 해석도 뒤집힌다"

**c=500 에서 stage4 > stage3 였고, c=1000 에서 stage3 > stage4 로 뒤집혔다.** 같은 시스템, 같은 데이터, 동시성만 달라졌는데 우열이 반전. 벤치 결과는 "X 가 Y 보다 빠르다" 가 아니라 **"조건 Z 에서 X 가 Y 보다 빠르다"** 로 기록해야 한다. 조건을 빼고 결론만 들고 다니면 거짓말이 된다.

### 10. "책의 진화 경로는 단일 머신 포화 이후에만 의미가 있다"

c=50 부터 c=1000 까지 전 동시성에서 **stage1 (HashMap 단일) 이 항상 최고 RPS** 였다. stage2/3/4 는 단 한 번도 개선이 아니었다. 이건 책이 틀렸다는 뜻이 아니다 — **단일 머신이 에러 0 으로 버티는 상황에서 그 다음 단계를 강제했기 때문** 이다. 책의 진화 경로는 **"단일 머신이 감당 못할 때에만 다음 단계로 넘어가라"** 는 암묵적 조건부 문장이다. 그 전제를 잊고 "박스를 많이 그린 설계가 좋은 설계" 라고 착각하기 쉽다.

### 11. "가설은 조건부로만 맞는다 — 일반화 금지"

세션 중 가설을 **3번 수정** 했다:

1. **초기**: "책의 진화 경로는 개선이다" → c=50 에서 전 단계 역방향, 기각
2. **수정 1**: "동시성을 올리면 stage3/4 가 역전할 것" → c=1000 에서도 stage1 이 최고, 기각  
3. **수정 2**: "풀은 어드미션 컨트롤이다" → stage2 에서 +21% 개선, 부분 기각. **"풀 ≪ downstream 용량 → 병목, 비슷 → 어드미션 컨트롤"** 로 재수정

**교훈**: 모든 가설은 "조건 X 에서 Y 가 관찰된다" 형태로 기록해야 한다. 조건을 빼고 "Y 는 항상 Z 다" 로 일반화하면 다음 실험에서 반드시 반례가 나온다. **직관은 실험 뒤에 오는 것이지, 앞에 오는 것이 아니다.**

### 12. "복잡한 시스템일수록 튜닝 효과가 작다"

stage2 (단일 replica, 단순) → pool 튜닝 +21%  
stage3 (LB + 2 replica, 복잡) → pool 튜닝 ±0%

복잡한 시스템은 병목이 여러 곳에 분산되어 각각이 약해지고, 전체를 끌어올리려면 **모든 박스를 동시에 튜닝** 해야 한다. 이게 "마이크로서비스는 운영이 힘들다" 의 작은 실험적 증거다. **"일단 단순하게 시작하고, 단순한 구조의 튜닝이 막힐 때에만 복잡도를 추가해라"** 는 실무 원칙이 데이터로 뒷받침됨.

### D2. HikariCP pool 확장을 별도 확장 실험으로 추가할지

- **선택지 B**: "다음에 시도할 것" 에 언급만 하고 넘어감
- **결정**: c=1000 재실행 결과를 보고 사용자와 재상의 (미결)

### D3. 캐시 실패 현상을 "실패" 로 기록할지 "조건부 이득" 으로 기록할지

- **상황**: c=50 에서 stage4 는 stage3 보다 느림. c=500 에서 stage4 는 stage3 보다 미세하게 빠름. c=1000 에서 다시 stage4 가 stage3 보다 느림
- **c=1000 추가 관찰 후 재선택**: "Redis 는 조건부 이득" 이라는 결론은 **c=500 일회성 관찰의 과잉 일반화** 였다. c=1000 에서 관계가 뒤집힌 걸 보면 stage3 과 stage4 는 본질적으로 **같은 병목** (downstream 에 매달린 400+ 스레드 경합) 을 살짝 다르게 표현할 뿐
- **최종 결론**: "이 워크로드·이 동시성·이 클라이언트 설정에서 Redis 는 지속적 이득이 아니다." 단일 측정치로 우열을 주장하지 않고, **측정 조건 전체를 결론에 포함시킴**
- **이유**: 책의 일반론 ("캐시 = 빠름") 을 무비판적으로 받아들이지 않는 것이 이 저장소의 목적. 측정은 특정 조건의 단면일 뿐

### D4. HikariCP 튜닝 실험 — 실행 및 결과

- **선택**: A. stage2/3/4 를 pool=100, c=1000 으로 재벤치
- **실행**: `bench-results/c1000-tuned/` 에 3 파일 생성
- **결과 요약**: 가설 조건부 지지
  - stage2: 가설 반례 (+21% RPS) — 단순 시스템에서 pool=20 은 하드 리밋
  - stage3: 가설 지지 (RPS 동일, p50 악화, histogram spread)
  - stage4: 혼합 (tail 개선, median 악화)
- **파생 결정**: 원 가설 폐기, **조건부 가설** 로 수정. pool 크기의 역할은 downstream 처리 능력과의 비율에 의존

### D5. 책 1장의 나머지 주제를 추가 실험으로 커버할지

- **상황**: 책 1장은 단일→DB→LB→캐시 외에도 **수직 확장, DB 다중화, 무상태 웹 계층, CDN, 메시지 큐, 로깅·메트릭, 샤딩** 을 언급. 우리 실험은 이 중 3~4개만 직접 구동
- **선택지**:
  - A. 커버되지 않은 주제도 추가 실험 (챕터 분량 커짐, 세션 길어짐)
  - B. 현재 실험으로 끝내되, **커버리지 표** 로 "무엇을 했고 무엇을 안 했는지" 명시. 안 다룬 주제는 이후 챕터들에서 자연스럽게 다뤄짐 (예: DB 샤딩은 `key-value-store` 챕터, 메시지 큐는 `notification-system` 챕터)
- **결정**: **B 권장**. scaling-foundations 는 "1장의 개념을 전부 구현" 이 아니라 "1장의 핵심 메시지(단계적 확장) 를 직접 감각" 하는 것이 목적. 남은 주제는 해당 주제를 주제로 하는 챕터에서 더 깊게 다루는 게 낫다. 사용자 결정 대기
