## 단계 1 — "절대 빼먹지 말 질문"

면접관이 자청하지 않으면 **이쪽에서 묻는다**. 뭐든 가정한 채 시작하지 말 것.

- **사용자**: 누가 쓰는가? B2C / B2B / 내부 도구? 동시 사용자 규모?
- **사용 패턴**: 읽기 vs 쓰기 비율은? 핫스팟이 있는가? (예: 톱 1% 가 트래픽의 90%)
- **규모**: DAU? 일 요청? 데이터 보관 기간? 1년 후? 5년 후?
- **지리**: 단일 리전인가, 글로벌인가?
- **일관성**: 강한 일관성 vs 결과적 일관성? 사용자가 쓴 데이터가 즉시 보여야 하는가?
- **가용성**: 99.9% 면 충분한가, 99.99% 가 필요한가?
- **지연시간**: p95 목표는?
- **명시적 비범위**: 인증·결제·관리자 콘솔 같은 것은 다루지 않는다고 합의

## 단계 2 — 개략적 설계 빌딩블록

대부분의 시스템은 이 박스들의 조합이다. 면접에서는 각 박스에 **이름과 책임 한 줄**을 쓰면 충분하다.

```
[client] → [DNS] → [CDN] → [LB] → [API gateway] → [stateless service]
                                                       ├→ [cache: Redis]
                                                       ├→ [primary DB]
                                                       ├→ [read replicas]
                                                       ├→ [search index]
                                                       ├→ [message queue] → [worker pool]
                                                       └→ [object store: S3]
                                                       
                                       [observability: metrics/logs/traces]
```

설계 단계에서 **반드시 결정**해야 할 것:

1. **API 모양**: REST? gRPC? GraphQL? WebSocket? 핵심 엔드포인트 3~5개
2. **데이터 모델**: 핵심 엔티티 + 관계. RDB vs NoSQL 선택 근거
3. **저장 전략**: 단일 DB → 읽기 복제 → 샤딩 → CQRS 중 어디까지?
4. **캐싱 위치**: client / CDN / API / DB read-through 중 어느 층?
5. **비동기 경계**: 어떤 작업이 동기 응답이고, 어떤 게 큐를 탈 수 있는가?

## 단계 3 — 상세 설계에서 주로 파게 되는 주제

면접관이 "이 부분 좀 더 깊이" 라고 했을 때 자주 등장:

| 주제 | 자주 묻는 질문 | 트레이드오프 축 |
|---|---|---|
| 캐싱 | TTL? Invalidation? Write-through vs Write-back? Stampede 방지? | 일관성 ↔ 지연시간 |
| 샤딩 | 키 선택, hot shard, resharding 비용 | 균등 분포 ↔ 운영 복잡도 |
| 복제 | 동기 vs 비동기, leader-less, quorum (R+W>N) | 일관성 ↔ 가용성 (CAP) |
| 큐 | At-least-once vs Exactly-once, ordering 보장, dead letter | 처리량 ↔ 의미론 강도 |
| 인덱싱 | B-tree vs LSM, 역색인 (검색) | 읽기 ↔ 쓰기 속도 |
| 카운터 | 정확 vs 근사 (HyperLogLog), 분산 증가 | 정확도 ↔ 처리량 |
| 고유 ID | UUID v4 vs Snowflake vs DB sequence | 정렬 가능성 ↔ 분산 친화성 |
| 알림 | 푸시 vs 풀, fan-out 위치 | 지연시간 ↔ 저장/대역폭 |
| 검색 | 정확 매치 vs 풀텍스트 vs 자동완성 | 색인 비용 ↔ 검색 품질 |

## 단계 4 — 마무리에서 언급하면 좋은 것

마무리는 1분 안에 끝나는 "이걸 빼먹지 마" 체크리스트.

- **병목 짚기**: 현재 설계의 첫 병목이 어디인가? (대개 DB write 또는 cache invalidation)
- **SPOF 짚기**: leader, primary DB, 단일 region 같은 단일 장애점
- **운영**: 메트릭·로그·트레이싱 어디에 붙일 것인가? 알람 임계값?
- **배포**: blue-green / canary / feature flag 중 무엇? rollback 시나리오
- **확장 경로**: "X10 트래픽이 오면 다음 조치는 ____"
- **비용**: 가장 큰 비용 항목이 무엇인가? (대개 egress 또는 DB)
- **다음 단계 가설**: 데이터가 더 있다면 무엇을 측정해보고 싶은가

## 안티패턴 (책 + 경험)

- **요구사항을 묻지 않고 바로 박스 그리기** — 면접관이 가장 싫어함
- **면접관의 힌트 무시** — 면접관이 "이 부분 더 자세히" 라고 하면 그게 채점 포인트
- **특정 기술 선택을 정당화 없이 단언** — "Cassandra 쓰면 됨" → 왜? 무엇과 비교했는가?
- **트레이드오프 없이 "최선" 만 말하기** — 모든 결정에는 비용이 있다
- **숫자 회피** — DAU·QPS·저장량을 입에 올리지 않으면 면접관은 "감 없는 사람" 으로 본다
- **운영을 잊기** — 동작은 하지만 "어떻게 죽고 어떻게 부활하는가" 가 빠진 설계
- **시간 관리 실패** — 1단계에 30분 쓰고 3·4단계를 못 마침. 시간 배분을 미리 선언
