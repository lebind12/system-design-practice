# 8장 — URL 단축기 책 정리

구현에 앞서 책 8장의 요지를 정리한다. 본 문서는 **책 요약** 과 **구현 시 마주칠 의사결정 포인트** 두 축으로 구성한다. 실제 구현·벤치·회고는 [README.md](README.md) 에 별도로 기록한다.

---

## 1. 문제 정의 (Step 1. 요구사항)

### 기능 요구사항
- 긴 URL → 짧은 URL 로 단축 (`POST /shorten`)
- 짧은 URL → 원본 URL 로 리다이렉트 (`GET /{shortKey}`)
- (옵션) 사용자 지정 별칭 / 만료 / 클릭 분석 등

### 비기능 요구사항
- **높은 가용성** — 단축 링크가 죽으면 모든 다운스트림 링크가 죽는다
- **확장성** — 읽기 경로가 쓰기보다 압도적으로 많은 트래픽
- **장애 내성** — 단일 노드 장애가 서비스 중단으로 이어지면 안 됨
- **예측 불가능성** — 짧은 키는 추측 불가능해야 함 (보안 · 열거 공격 방지)

### 책이 가정한 규모
| 항목 | 수치 |
|---|---|
| 쓰기 | 1억 건/일 → 약 **1,160 writes/sec** |
| 읽기 | 쓰기의 10배 → 약 **11,600 reads/sec** |
| 10년 누적 레코드 | 3,650억 건 |
| 평균 URL 길이 | 100 바이트 |
| 10년 누적 저장소 | 약 36.5 TB |

> **의사결정 포인트 ①** — 내 실험 저장소에서는 이 수치를 그대로 가정하지 않는다. 로컬 단일 노드에서 의미 있는 측정을 얻으려면 규모를 1~2 자리 축소한 "소규모 시나리오" 로 번역해야 하며, 그때도 **읽기:쓰기 = 10:1 비율** 은 유지한다. (이 비율이 캐시 효과와 키 생성 전략 선택에 결정적)

### 1-a. 스케일 결정 — 왜 책의 1/100 인가, 그리고 그 한계

책의 11,600 rps 는 단일 노드의 한계가 아니라 **분산 패턴(샤딩·복제·LB)을 강제하기 위한 가정값**이다. 실제로 노트북 한 대의 단일 Spring Boot + PostgreSQL 도 적절한 워크로드라면 1,000+ rps 를 무리 없이 처리한다. 따라서 이 챕터의 목표는 "책의 11,600 rps 를 단일 노드로 재현" 이 아니라 **"책 스케일의 1/100 지점에서, 단일 노드 변형들의 상대적 차이를 측정"** 하는 것이다.

이 결정에서 솔직해야 할 두 가지가 있다.

1. **단일 노드는 1/100 부하(≈100 rps) 에 대해 시시할 정도로 여유롭다.** 그 부하만 측정하면 캐시 효과가 노이즈에 묻혀 안 보인다.
2. **책 스케일로의 외삽은 선형 가정이 거의 항상 틀린다.** 1/100 에서 잰 단위 시간을 100배 곱해 "책 스케일에서는 X rps" 같은 결론을 내면 안 된다 (커넥션 풀, 디스크 IO, GC 가 비선형).

이 두 가지를 인정하면서도 의미 있는 측정을 얻기 위해, 벤치 전략을 **두 축의 결합** 으로 설계한다.

| 축 | 무엇을 보는가 | 출력 |
|---|---|---|
| **A. 고정 부하 비교** | 책 1/100 부하(≈100 rps, c=50) 에서 각 변형의 p50/p95/p99 를 같은 표에 나란히. _"같은 부하에서 무엇이 변하는가."_ | 시나리오 × 동시성 표 한 행 |
| **B. 포화 곡선** | 같은 변형들을 c=10/50/100/200/500 로 스윕해 단일 노드의 포화 지점을 비교. _"단일 노드가 어디서 무너지고, 어떤 변경이 그 한계를 어디로 옮기는가."_ | 시나리오 × 동시성 표 전체 + RPS·p95 차트 |

A 와 B 는 같은 데이터셋(`bench-results/c<N>/<scenario>.txt`) 에서 동시에 나온다 — 한 번의 스윕으로 두 표가 함께 채워진다. 따라서 운영 비용은 사실상 B 한 번이고, A 는 B 의 한 행을 추출해서 보는 관점일 뿐이다.

> **결론적 프레이밍** — _"책의 11,600 rps 는 분산을 강제하기 위한 숫자다. 이 챕터는 그 가정의 1/100 지점에서 단일 노드 변형들의 상대적 차이를 측정한다. 외삽 대신 두 가지를 동시에 본다: (A) 같은 부하에서 무엇이 변하는가, (B) 단일 노드가 어디서 무너지는가."_

## 2. 개략적 추정 (Step 2. Back-of-envelope)

책은 다음 계산을 안내한다:

```
초당 쓰기     = 1억 / 86,400초 ≈ 1,160/s
초당 읽기     = 1,160 × 10 ≈ 11,600/s
10년 레코드   = 1억 × 365 × 10 ≈ 365억  ← (책은 3,650억으로 기재; 판본에 따라 상이)
키 공간 요건   = 62^n ≥ 최대 레코드 수
  → n=7 → 62^7 ≈ 3.5조  (10년치를 거뜬히 수용)
```

### 짧은 키 길이 산정
- **Base62 (0-9, a-z, A-Z) 7자리** → 약 3.5조 개. 10년 운영에도 여유
- 6자리는 약 568억 → 판본에 따라 부족할 수 있음
- 8자리 이상은 시각적으로 "짧지 않다"는 인상

> **의사결정 포인트 ②** — 내 구현은 7자리 Base62 를 채택한다. 이유: (1) 학습 목적상 책의 기본값을 재현 (2) 7자리가 실제 서비스(bit.ly 등)의 길이 감각과도 가까움.

## 3. 상위 설계 (Step 3. High-level design)

### 3-1. API 엔드포인트
책은 두 개의 엔드포인트만 제시한다.

| 메서드 | 경로 | 동작 |
|---|---|---|
| `POST` | `/api/v1/data/shorten` | body `{longUrl}` → `{shortUrl}` |
| `GET` | `/api/v1/{shortUrl}` | 301/302 리다이렉트 |

### 3-2. 301 vs 302 — 책이 강조하는 핵심 트레이드오프
| | 301 Moved Permanently | 302 Found (Temporary) |
|---|---|---|
| 브라우저 캐싱 | **강하게 캐싱** (이후 요청은 서버 안 탐) | 캐싱 약함 (매 클릭마다 서버) |
| 서버 부하 | 낮음 — 같은 링크 반복 클릭이 DB 에 도달하지 않음 | 높음 — 모든 클릭이 서버로 옴 |
| 클릭 분석 | **불가능** (서버가 요청을 못 봄) | **가능** — 모든 클릭 집계 가능 |

> **의사결정 포인트 ③** — "서버 부하 최소화"와 "클릭 분석" 사이 선택. 책은 분석 수집을 위해 **302** 를 자주 쓴다고 설명한다. 내 구현에서도 읽기 경로 벤치마크 의미를 살리려면 **302** 가 적절 — 301 이면 캐시가 단 한 번만 히트하고 이후는 측정 불가.

### 3-3. URL 단축 알고리즘 — 두 접근
책이 제시하는 두 가지 방식과 장단점.

#### (A) Hash + Collision Resolution
1. `longURL` 을 MD5/SHA-1 등으로 해시
2. 앞 7자 추출 → `shortKey` 후보
3. DB 에서 충돌 검사
4. 충돌이면 `longURL || predefined_string` 을 다시 해시하고 반복

**장점**
- 외부 의존 없음 (unique ID 생성기 불필요)
- 같은 `longURL` → 같은 `shortKey` 가 자연스럽게 성립 (멱등)

**단점**
- **충돌 검사 = 매 쓰기마다 DB 조회** → 쓰기 경로 지연
- 충돌 확률이 높아질수록 재시도 증가 → p99 꼬리 지연 악화

#### (B) Base62 Conversion (카운터 기반)
1. **Unique ID 생성기** (예: 분산 Snowflake — 7장 주제) 로 유일 ID 획득
2. ID 를 Base62 로 인코딩 → `shortKey`
3. DB 에 저장

**장점**
- ID 가 유일하므로 **충돌 조회 불필요** → 쓰기 경로 단순·빠름
- Base62 길이는 ID 가 커질수록 증가 (초기에는 짧고 점차 길어짐)

**단점**
- **외부 unique ID 생성기 의존** (7장 선행 구현 필요 또는 대체 방안)
- ID 가 **예측 가능**할 경우 열거 공격 우려 → 난수화 트릭 필요
- 같은 `longURL` 이 들어와도 매번 새 `shortKey` 가 만들어짐 (중복 URL 처리 별도)

> **의사결정 포인트 ④** — 책은 (B) Base62 를 권장한다. 이유: 쓰기 경로 단순성과 충돌 제거. 내 구현도 **(B) Base62** 를 기본으로 하되, **확장 실험에서 (A) Hash 방식으로 바꿔 p99 쓰기 지연 차이를 측정** 하는 것이 하나의 스토리가 된다. 또한 7장 (unique-id-generator) 을 아직 구현하지 않았으므로, **로컬 unique ID 는 DB auto-increment 또는 `AtomicLong`** 으로 대체한다 — 분산 ID 생성기는 7장 챕터에서 별도로 다룬다.

### 3-4. 데이터 모델
```
urls (
  id        BIGSERIAL  PRIMARY KEY,
  short_key VARCHAR(7) UNIQUE NOT NULL,
  long_url  TEXT       NOT NULL,
  created_at TIMESTAMP
)
```
- `id` → Base62 변환의 원천
- `short_key` 에 **unique index** — 읽기 경로의 주 조회 키
- `long_url` 중복 처리는 **Optional** (책은 언급하되 필수로 강제하지 않음)

> **의사결정 포인트 ⑤** — `long_url` 에 index 를 걸 것인가? 걸면 "같은 URL 재요청 시 기존 키 재사용" 이 가능해지지만 쓰기마다 조회가 발생. 책은 이를 "선택적"이라 언급. 내 실험에서는 **기본은 걸지 않음** (쓰기 단순화), **확장 실험으로 Bloom filter + DB 조회** 를 추가해 쓰기 경로 지연 변화 측정.

## 4. 상세 설계 (Step 4. Deep dive)

### 4-1. URL 단축 플로우 (쓰기 경로)
```
client → POST /shorten {longUrl}
  ↓
  (옵션) longUrl 중복 검사  ── 있으면 기존 shortKey 반환
  ↓
  unique ID 생성 (auto-increment or 분산 ID)
  ↓
  base62(id) → shortKey
  ↓
  DB INSERT (id, shortKey, longUrl)
  ↓
client ← {shortUrl}
```

### 4-2. URL 리다이렉트 플로우 (읽기 경로)
```
client → GET /{shortKey}
  ↓
  LB → web server
  ↓
  Redis GET shortKey
    ├─ HIT  → longUrl 반환
    └─ MISS → DB SELECT long_url WHERE short_key=?
                  ├─ 없음 → 404
                  └─ 있음 → Redis SET + longUrl 반환
  ↓
client ← 302 Location: {longUrl}
```

> **의사결정 포인트 ⑥** — **캐시 없는 MVP 와 캐시 있는 확장 버전의 p99 차이** 가 이 챕터의 핵심 실험 포인트. 책은 캐시를 당연시하지만, 실측 수치로 그 효과를 보는 것이 이 저장소의 가치다.

### 4-3. Base62 인코딩 — 구현 상세
```
문자집합: 0-9 a-z A-Z  (총 62자)

function toBase62(n: long): string
  result = ""
  while n > 0:
    result = charset[n % 62] + result
    n = n / 62
  return result

function fromBase62(s: string): long
  n = 0
  for c in s:
    n = n * 62 + charset.indexOf(c)
  return n
```

- id=11157 → "2TX"
- id=11,157,000,000 → "C73BG7"
- 7자리 = 62^7 ≈ 3.5조

### 4-4. 웹 서버 스케일링 · DB 스케일링
책은 마지막 deep dive 에서 다음을 언급하지만 **스케치 수준**으로만 다룬다:
- **웹 서버** — stateless 이므로 수평 확장 용이, LB 뒤에 N 대
- **DB** — 쓰기가 감당 안 되면 샤딩 (short_key 기준 해시 샤딩)
- **캐시** — LRU 정책, TTL, cache stampede 방지 (책은 상세히 다루지 않음)

> **의사결정 포인트 ⑦** — 내 로컬 환경에서는 DB 샤딩까지는 실험하지 않는다. 단일 PostgreSQL + 캐시 유무 비교에 집중한다. 수평 확장은 "MVP 앱 단일 노드 vs 앱 2 노드 + LB" 정도를 고려할 수 있으나 우선순위는 낮다.

## 5. 마무리 (Step 5. Wrap up)
책이 wrap-up 에서 언급하는 추가 고려사항:
- **Rate limiter** (4장) — 악성 사용자의 대량 단축 요청 방지
- **웹 서버 확장** — stateless + LB
- **DB 확장** — 샤딩, 복제
- **분석 / Analytics** — 클릭 통계 수집 (302 리다이렉트 전제)
- **가용성 · 일관성 · 신뢰성**

> **의사결정 포인트 ⑧** — rate limiter 는 2번 챕터 (`rate-limiter`) 에서 별도로 만든 뒤 이 챕터로 역수입할 수도 있다. 현재 챕터 1 (이 챕터) 구현에는 포함하지 않음.

| ① | 가정 트래픽 | 1,160 wps / 11,600 rps | 축소 시나리오 (예: 100/1000 rps) | 로컬 단일 노드 측정 가능 범위 |

| ④ | 단축 알고리즘 | Base62 권장 | **MVP: Base62 (DB auto-increment 기반)**, **실험: Hash + 충돌검사 비교** | 쓰기 경로 지연 차이 관찰 |
| ⑤ | long_url 중복 처리 | 선택적 | **MVP: 미처리**, **실험: Bloom filter** | 쓰기 경로 오버헤드 관찰 |
| ⑥ | 캐시 레이어 | 당연히 사용 | **MVP: 없음**, **실험: Redis** | 읽기 p99 차이 정량화 |
| ⑦ | DB/앱 수평 확장 | 후반에 언급 | 미실험 (우선순위 낮음) | 로컬 단일 노드 집중 |
| ⑧ | Rate limiter | wrap-up 언급 | 2번 챕터에서 분리 | 관심사 분리 |

## 7. 이번 챕터의 실험 스토리 (요약)

**MVP** — Spring Boot + PostgreSQL, Base62(DB auto-increment), 302 리다이렉트, **캐시 없음**

**확장 실험 (제안)**
1. **Redis 캐시 추가** — 읽기 경로 p50/p95/p99 비교, cache hit ratio 측정
2. **Hash+충돌검사 방식** 으로 쓰기 경로 교체 — Base62 대비 p99 쓰기 지연 증가량 관찰
3. **Bloom filter 로 long_url 중복 검사** — 쓰기 처리량 변화 관찰

이 셋이 완료되면 "왜 책이 Base62 + 캐시를 기본으로 권장하는지" 를 수치로 설명할 수 있다.
